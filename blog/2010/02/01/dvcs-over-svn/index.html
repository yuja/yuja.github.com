Git と Mercurial が Subversion より優れている点

2010 年 2 月 1 日

For home projects, can Mercurial or Git (or other DVCS) provide more advantages over Subversion? - Stack Overflow より、抄訳。

この記事は Git と Subversion の比較だけど、ここで言っていることは全て Mercurial にもあてはまる。

リポジトリの初期化:

  • git:

      git init

  • subversion:

      svnadmin create /path/to/repo
  svn import http://long.url.to/repo yourwork
  rm -R yourwork
  svn checkout http://long.url.to/repo yourwork

うぁー、めんどくせー!!

リポジトリとメタデータ:

  • git: リポジトリとメタデータを、最上位の .git ディレクトリに保管。
  • subversion: リポジトリは別の場所に。メタデータは 各ディレクトリ の .svn に保管。

けっこうウザい。

リビジョンの名前・番号:

  • git: グローバルにユニークなハッシュ値と、特定リビジョンからの相対値。
  • subversion: リビジョン番号。

Subversion のほうがパッと見で分かりやすいね。

ブランチとマージ:

  • git: ブランチの作成とマージは簡単。だって分散バージョン管理はそう設計されているんだから。リネームしても大丈夫。
  • subversion(<1.5): ブランチの作成は簡単。マージは死ぬ。
  • subversion(>=1.5): svn:mergeinfo があるけど、 Git より複雑。単純な場合しか対処できない。リネームしたらアウト。

タグ:

  • git: タグは不変。 (訳注) タグのつけ直しは可能。
  • subversion: ブランチと同じ。(タグった後で)変更可能。

この設計は明らかにクソだ。

キーワード展開:

  • git: かなり限定されている。だって変更はファイルベースじゃないし。
  • subversion: (訳注) CVS レベルのサポートあり。 Subversion はファイルベースです。

バイナリファイル:

  • git: GNU diff と同じアルゴリズム。(訳注) '\0' 検出だと思う。
  • subversion: svn:mime-type による。

無視ファイル:

  • git: バージョン管理された .gitignore ファイルによる。
  • subversion: ディレクトリごと に svn:ignore を設定。

何でこうも Subversion は面倒くさいんだろうね。

コミットを修正:

  • git: 公開していないコミットならどんと来い。標準でサポートしてるぜ。
  • subversion: コミットメッセージだけならプロパティの修正でできなくない。(訳注) 基本的にダメ。

ツール:

  • git: bisect (バグの2分探索) みたいな便利なコマンドがある。
  • subversion: 古いから周辺ツールがいろいろある。特に MS Windows なら。

Subversion の GUI はさすがだ。
