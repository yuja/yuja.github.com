Subversion と Git/Mercurial/Bazaar は全く別物

2010 年 2 月 23 日

Popularity of Git/Mercurial/Bazaar vs. which to recommend - Stack Overflow より。

they are different in their nature.

  • SVN is a REVISION system (it stores branch and tag through cheap copy only! Merge support is not very efficient), and it is centralized.
  • Mercurial or bazaar are FILE VCS (they store versions of files), and distributed
  • Git, and that is very hard to grasp, is a CONTENT VCS (it stores delta of content, not the file itself: two files with the same content will be stored only once)

そいつらは全く別物なんだ。

  • SVN は「リビジョン」システムで、集中型だ。 (ブランチもタグもみみっちぃコピーでしかない! し、 マージサポートは貧弱だ。)
  • Mercurial と bazaar は「ファイル」 VCS で、分散型だ。(ファイル のバージョンを保持している。)
  • Git はちょっと説明しづらいけど、 コンテンツ VCS なんだ。 (ファイルじゃなくて コンテンツ の差分を保持している。つまり、同じ内容(コンテンツ)のファイルが 2 つあると、 1 度だけ保存される。)

(訳注) Mercurial/Bazaar がファイルごとに変更を追跡しているのに対し、 Git はコンテンツ全体を追跡しているということ。 Mercurial/Bazaar が RCS のようなファイルベースの VCS だということでは決してない。

コメントにあるように、 Mercurial と Git の間に大した差は無いと思う。

That means:

  • if you have a simple merge workflow, in a single development location, stick with SVN
  • if you have several development places, a distributed VCS is more adapted.
  • if you have a complex merge workflow, any modern VCS is better than SVN which struggle to keep merge informations at the right places for years. It then depends on the tools (Mercurial has a more advanced Windows support for instance)
  • if you have mainly text file and not too-large binary files, Git is excellent, provided you are aware of its limits.

ということで、

  • 単純な マージワークフロー で開発場所がひとつなら、 SVN で押し通そう。
  • 開発場所がいくつかあるなら、分散 VCS のほうが適しているだろう。
  • 複雑なマージワークフロー が必要なら、現代の VCS はいずれも SVN より優れている。 SVN では、マージ情報を適切に維持して行くのに四苦八苦するだろう。 選択はツールによる。(例えば、 Mercurial の Windows サポートは他よりいい。)
  • 対象が主にテキストファイルとそれほどデカくないバイナリファイルであれば、 Git はかなり良いよ。 その制限 を理解した上でね。

(訳注) Mercurial もデカいファイルの扱いは上手くない。 管理対象が主にバイナリファイルなら、 SVN が良いだろう。バイナリファイルはマージできないので、分散型は向いていない。

分散型と集中型の違いがよく分からないって? 分散バージョン管理入門 (イラスト入り) を読もう。
